# Yukti Security Audit — 2026-04-19

**Scope**: full P(-1) scaffold-hardening pass against Yukti 1.3.0, targeting
the 2.0.0 release.
**Auditor**: internal.
**Target release**: 2.0.0 (post-fix).
**Method**: code walk of every `src/*.cyr` module + external CVE sweep of
adjacent kernel / util-linux / udev / partition-table surfaces (2024–2026).

## Summary

| Severity  | Count | Disposition                             |
|-----------|-------|-----------------------------------------|
| CRITICAL  | 0     | —                                       |
| HIGH      | 2     | Fix required before 2.0.0               |
| MEDIUM    | 5     | Fix required before 2.0.0               |
| LOW       | 6     | Schedule; defense-in-depth fixes        |
| Total     | 13    |                                         |

Yukti has **no memory-unsafe language primitives** (no raw pointer
arithmetic in user code; struct access is `load64`/`store64` with
named offsets) and **no libc**. The attack surface is narrow: parsers
(`/proc/mounts`, uevent, MBR/GPT, CDROM TOC), syscall wrappers
(mount/umount/eject/ioctl), and SQL-like queries to the embedded
`patra` store.

The two HIGH findings are both physically-adjacent-attacker scenarios
(malicious USB or disk insert). Both have contained blast radius (DB
tampering, stack corruption in a non-privileged parser path) but
should be fixed before 2.0.0 promotes the kernel-safe split that
widens Yukti's deployment surface.

---

## Methodology

1. **Attack-surface enumeration**. Identified every external input
   boundary:
   - `parse_uevent()` (netlink, fuzzed)
   - `/proc/mounts` parser (fuzzed)
   - MBR + GPT partition tables (**not fuzzed** — gap)
   - sysfs attributes (kernel-provided, lower trust weight)
   - CDROM TOC via ioctl (drive-provided)
   - udev rule fields (user-provided)
   - mount point / fs type strings (user-provided)
   - Network share host / path / credentials
2. **Code walk** per surface with checklist: input validation, buffer
   safety, syscall return handling, pointer arithmetic, command
   injection, path traversal, integer overflow.
3. **CVE sweep** of the last 24 months in adjacent kernel / util-linux
   / udev / GPT parser domains. Cross-referenced each applicable class
   against Yukti.
4. **Findings triaged** with severity per CVSS-style guidance
   (CRITICAL / HIGH / MEDIUM / LOW) matching the severity ladder in
   `CLAUDE.md`.

## Findings

### [HIGH-1] SQL injection via malicious USB device metadata

- **File**: `src/device_db.cyr`
- **Lines**: 113–115, 126–130, 139–151, 178–190, 195–202, 217–220,
  224–235, 246–250
- **Class**: CWE-89 (improper neutralization of SQL)
- **CVE analogs**: general SQL injection class; not a direct kernel CVE
  match but the pattern mirrors dozens of application-layer CVEs in
  2025–2026 (e.g. CVE-2025-15036, CVE-2026-30848).

**Description**. Every `patra_exec` / `patra_query` call in
`device_db.cyr` builds the SQL statement by concatenating strings with
attacker-influenceable values. The values — `serial` (from `ID_SERIAL`
udev property), `vendor`, `model`, `mount_point`, `fs_type_str`,
`dev_path` — all flow from USB descriptor fields or mount-time
arguments. A USB stick that advertises `ID_SERIAL` as
`'); DROP TABLE devices; --` (or similar) injects SQL into the
INSERT/UPDATE/DELETE issued on device arrival.

**Impact**. Tamper / delete the yukti device-history DB; poison mount
history for other devices; if patra supports attached subqueries,
potentially exfiltrate preferences of other devices. No privilege
escalation outside the DB sandbox, but the DB is a trust boundary
consumers (argonaut automount) read back from.

**Proof pattern**:

```
udev ACTION=add DEVPATH=/devices/… ID_SERIAL=abc'); DROP TABLE devices; --
  → parse_uevent → di_set_serial(info, "abc'); DROP TABLE devices; --")
  → device_db_record_seen(info)
  → patra_exec(..., "INSERT INTO devices VALUES ('abc'); DROP TABLE devices; --', ...)")
```

**Mitigation**. Parameterize via `patra_bind_*` (if patra exposes
bind API) or escape single quotes (`'` → `''`) before interpolation,
and reject strings containing NUL bytes. Preferred: switch all
`patra_exec` call sites to prepared-statement form.

### [HIGH-2] Unbounded GPT `entry_size` → stack buffer overflow

- **File**: `src/partition.cyr`
- **Lines**: 356–363 (`_parse_gpt_entries`), 487–495 (caller)
- **Class**: CWE-121 (stack-based buffer overflow)
- **CVE analogs**: CVE-2026-23474 (RedBoot partition parser), general
  class of unbounded-entry-size parser bugs.

**Description**. The GPT header's `entry_size` field (u32 at offset
84 of the header) is read from the on-disk GPT header without an
upper bound:

```cyrius
var entry_size = load32(&gpt_hdr + 84);
if (entry_size == 0) { entry_size = GPT_ENTRY_SIZE; }     # only lower bound
if (num_entries > GPT_MAX_ENTRIES) { num_entries = GPT_MAX_ENTRIES; }
```

Then `_parse_gpt_entries` allocates a fixed-size stack buffer and
passes `entry_size` to `_read_at` as the byte count to read:

```cyrius
fn _parse_gpt_entries(fd, entry_start_lba, num_entries, entry_size) {
    var buf[128];                                          # 128-byte stack buf
    ...
    var n = _read_at(fd, byte_offset + i * entry_size, &buf, entry_size);
    ...
}
```

A malicious disk with `entry_size = 4096` (or any value > 128) causes
the kernel's `read(2)` to write up to `entry_size` bytes into a
128-byte stack buffer. Adjacent stack frame (saved RBP / return
address on x86_64) is corrupted.

**Impact**. Local code execution in the parent of `read_partition_table`,
triggered by physical USB insert. Yukti has no stack canaries (raw
cc5 codegen); no mitigation other than not overflowing.

**Mitigation**. Clamp `entry_size` to `[GPT_ENTRY_SIZE, buf_size]`
(i.e. exactly 128 bytes — the spec requires 128-byte multiples but
Linux kernel and mainstream tooling only emit 128; larger sizes are
vendor extensions we don't parse). Reject GPT headers with
`entry_size > 128` outright.

### [MED-1] `_is_forbidden_mount` blacklist is incomplete + not canonicalized

- **File**: `src/storage.cyr`
- **Lines**: 217–228 (blacklist), 346–363 (validator)
- **Class**: CWE-22 (path traversal) / CWE-693 (incomplete enum)

**Description**. The forbidden-mount list covers `/`, `/bin`, `/sbin`,
`/usr`, `/etc`, `/boot`, `/sys`, `/proc`, `/dev` via exact `streq`.
Gaps:

- **Missing entries**: `/var`, `/root`, `/home`, `/lib`, `/lib64`,
  `/srv`, `/opt`, `/tmp`, `/mnt` (mounting over an existing mount).
- **No prefix match**: `/etc/cron.d` is not blocked; only exact
  `/etc`.
- **No canonicalization**: `/mnt/../etc` bypasses string compare
  because it's not literally `/etc`. No `realpath()` resolution.
- **Symlink traversal**: if `/run/media/SDA1` is a symlink to `/etc`
  placed by an unprivileged user before yukti reaches the validator,
  the string check passes.

**Impact**. If a caller with `CAP_SYS_ADMIN` (which mount already
requires) accepts arbitrary mount points from a lower-privileged
source, the gaps allow mounting over sensitive directories, shadowing
`/home`, `/var/log`, `/root`, etc.

**Mitigation**. (a) extend the literal blacklist; (b) add a prefix
check (`startswith("/bin/")`, etc.); (c) canonicalize via
`readlinkat()` loop or reject paths containing `..` and `//`;
(d) for known-good paths, use `openat(dirfd, component, O_NOFOLLOW)`
to refuse symlinks.

### [MED-2] Mount-path TOCTOU (CVE-2026-27456 class)

- **File**: `src/storage.cyr`
- **Lines**: 450 (`validate_mount_point`) then 467 or 483 (`syscall(165, …)`)
- **Class**: CWE-367 (TOCTOU)
- **CVE analog**: CVE-2026-27456 — util-linux mount TOCTOU, loop device
  swap between validate-and-use.

**Description**. `validate_mount_point()` checks the string form of
`mount_cstr`, then `storage_mount()` issues `syscall(165, dev,
mount_cstr, fs, flags, 0)`. Between the two, an attacker who can
create entries under the mount root (commonly `/run/media/` which is
world-writable in many configurations) can substitute a symlink that
resolves to `/etc` or similar.

**Impact**. Same as MED-1 but via a race rather than a blacklist gap.

**Mitigation**. Validate by `stat()`-ing the final component after
`mkdir()` with `O_NOFOLLOW`; reject if it's not a real empty
directory. Or use `open_tree(2)` + `move_mount(2)` (Linux 5.2+) to
pin the target fd before mount.

### [MED-3] `/proc/mounts` parser truncates at 8 KB

- **File**: `src/storage.cyr`
- **Lines**: 298–303 (`find_mount_point`)
- **Class**: CWE-120 (buffer copy without bounds check on total size)
  — not a classical overflow (we allocate enough), but a **silent
  truncation** that gives wrong answers.

**Description**.

```cyrius
var readbuf[8192];
var nbytes = file_read_all("/proc/mounts", &readbuf, 8192);
```

On container hosts, snap-heavy systems, or btrfs setups with many
subvolumes, `/proc/mounts` routinely exceeds 16–32 KB. `file_read_all`
stops at 8 KB, the trailing mounts are never seen, and
`is_mounted()` returns false for legitimately mounted devices.

**Impact**. Incorrect "not mounted" decision → yukti consumers
(argonaut automount) try to mount over an existing mount, or skip
eject safety unmount. Observable correctness bug, indirectly a
safety issue.

**Mitigation**. Read into a growable buffer, or read in chunks until
EOF. Add a test with a synthetic mount table >= 16 KB.

### [MED-4] Netlink uevent spoofing surface (CVE-2009-1185 class)

- **File**: `src/udev.cyr`
- **Lines**: 594–628 (monitor setup), 647 (`recvfrom`)
- **Class**: CWE-345 (insufficient verification of data authenticity)
- **CVE analog**: CVE-2009-1185 — udev unconditionally trusted
  netlink uevents; fix added origin check.

**Description**. `udev_monitor_with_filter` binds a `NETLINK_KOBJECT_UEVENT`
socket to multicast group 1 (kernel broadcasts). Modern kernels
(≥ Linux 3.16) restrict sends to group 1 to `CAP_NET_ADMIN`, so on
hardened kernels the attack is not reachable by unprivileged local
users. However:

```cyrius
var n = syscall(SYS_RECVFROM, fd, &buf, 8192, 0, 0, 0);
```

The `src_addr`/`addrlen` arguments (positions 5–6 of `recvfrom`) are
NULL — Yukti never reads `nl_pid` to confirm the sender is the
kernel (`nl_pid == 0`). On misconfigured kernels, or under
`unshare --net` scenarios where group restrictions differ, an
unprivileged process could inject forged uevents.

**Impact**. Forged "device attach" events with attacker-chosen
ID_SERIAL / DEVNAME feed directly into the SQL injection path
(HIGH-1) and the mount path (via argonaut automount consumer).

**Mitigation**. Provide `sockaddr_nl` for `msg_name`, check
`nl_pid == 0`, discard otherwise. Cheap defense-in-depth.

### [MED-5] Broken `run()` pattern in `udev_rules.cyr`

- **File**: `src/udev_rules.cyr`, `lib/process.cyr`
- **Lines**: `udev_rules.cyr:190, 210, 226, 228`, `lib/process.cyr:6–17`
- **Class**: functional bug with latent security implications.

**Description**.

```cyrius
str_builder_add_cstr(sb, "trigger --action=change ");
str_builder_add_cstr(sb, syspath_cstr);
var args = str_builder_build(sb);
var r = run("udevadm", str_cstr(args), 0);
```

`run()` performs `sys_execve("udevadm", argv=["udevadm", <concatenated
string>, NULL], envp=[NULL])`. Two problems:

1. `sys_execve()` does **not** do `PATH` lookup — `"udevadm"`
   resolves to the current working directory only. Every caller
   (`reload_rules`, `trigger_device`, `query_device`, `list_devices`)
   fails with `ENOENT`. These functions are currently non-functional.
2. Multi-token args are packed into a single argv slot, so even with
   an absolute path like `/bin/udevadm`, `argv[1]` would be the
   literal string `"trigger --action=change /sys/…"` — udevadm
   rejects.

**Security implication**: not currently exploitable (functions don't
run). If a future refactor introduces `execvp`-like PATH lookup or
shell fallback without fixing the tokenization, the concatenated
string becomes a shell/argv injection vector via crafted `syspath_cstr`.

**Mitigation**. Use `exec_vec()` with a proper argv vec: push
`"/usr/bin/udevadm"`, `"trigger"`, `"--action=change"`,
`"--path"`, `syspath_cstr` as separate elements. Add a test that
exercises each caller on a machine with udevadm installed.

### [LOW-1] `/sys/block/<base>/device/delete` write without O_NOFOLLOW

- **File**: `src/storage.cyr`
- **Lines**: 606–618
- **Class**: CWE-59 (link-following), defense-in-depth.

**Description**. Path constructed via string concat; opened with
`SYS_OPEN` (no `O_NOFOLLOW`). `base_name` is always a basename
substring in practice, but a symlink planted at
`/sys/block/<base>/device/delete` (requires root) would be followed.

**Mitigation**. `openat(AT_FDCWD, "/sys/block", O_PATH | O_DIRECTORY)`
then `openat(dirfd, "<base>/device/delete", O_WRONLY | O_NOFOLLOW)`.

### [LOW-2] TOC integer overflow on crafted lead-out

- **File**: `src/optical.cyr`
- **Lines**: 381 (`length * 100`), 389 (`leadout_lba * 2048`)
- **Class**: CWE-190 (integer overflow).

**Description**. `length` and `leadout_lba` are i32 values from
`CDROMREADTOCENTRY`. `length * 100` and `leadout_lba * 2048` can
wrap on crafted discs. Stored in i64 fields but the multiplication
itself is unchecked.

**Impact**. Displayed size / duration wrong on crafted discs. No
crash; values are just stored.

**Mitigation**. Cast to i64 before multiplying
(`(length cast i64) * 100`). Reject TOC entries with
`length > MAX_SECTORS_PER_TRACK`.

### [LOW-4] `default_mount_point` trusts label without length cap

- **File**: `src/storage.cyr:309–331`
- **Class**: CWE-20 (improper input validation).

**Description**. A USB label up to kernel limits (e.g. 512 bytes on
some FSes) is sanitized char-by-char but never truncated. Yukti's
consumers may allocate `/run/media/<label>` path buffers assuming a

## Remediation Plan

Fix order (and what each unlocks):

1. **HIGH-1** — parameterize or escape all `patra_exec` / `patra_query`
   call sites. Add a fuzz target that seeds `ID_SERIAL` with SQL
   metacharacters and asserts DB table integrity after `device_db_*`
   operations.
2. **HIGH-2** — clamp `entry_size` to `[128, 128]` in
   `read_partition_table`; reject headers otherwise. Add a GPT fuzz
   target (this is a coverage gap noted in the internal review — we
   fuzz uevent and mount table but not MBR/GPT).
3. **MED-1** — extend `_is_forbidden_mount` to cover
   `/var /root /home /lib /lib64 /srv /opt /tmp /mnt`, add prefix
   matching, reject `..` / `//` in the path. Unit tests per entry.
4. **MED-2** — TOCTOU fix via `O_NOFOLLOW` on a `stat` of the final
   mount component after `mkdir`.
5. **MED-3** — make `find_mount_point` read `/proc/mounts` in a loop
   until EOF (growable buffer). Test with synthetic ≥ 32 KB input.
6. **MED-4** — pass `sockaddr_nl` to `recvfrom`, enforce `nl_pid == 0`.
7. **MED-5** — switch udevadm callers to `exec_vec` with absolute
   paths and proper argv tokens.
8. **LOW-1 … LOW-6** — schedule individually; none block 2.0.0.

**Gates added to CI as part of this audit** (already landed in 1.3.0
workflow rewrite):

- `cyrius lint` fails build on any warning.
- `cyrius fmt --check` diff-clean required.
- Security scan: raw `execve`/`fork`/`sys_system`, `/etc /bin /sbin`
  path writes, oversized stack buffers (≥ 64 KB) are all failure
  conditions.
- Kernel-safe tripwire (`programs/core_smoke.cyr`) required green.
- Both dist bundles (full + core) regenerated and compared to
  committed copies.

## Non-findings (verified clean)

- **No `unsafe` / memory-unsafe primitives** — Cyrius has none; struct
  access is via `load64`/`store64` with named offset constants.
- **No libc, no FFI** — all syscalls direct, arity-checked by the
  compiler.
- **No `sys_system()` in `src/`** — verified by grep.
- **No raw `execve(59)` or `fork(57)` in `src/`** — verified. The only
  execve site is `lib/process.cyr` which is stdlib (audited separately).
- **No `/etc`, `/bin`, `/sbin` writes** — verified; only reads for mount
  blacklist.
- **Include discipline** — domain modules have zero transitive includes.
  Can't leak stdlib into kernel-safe subset.
- **Kernel-safe invariant** — `dist/yukti-core.cyr` verified zero
  `alloc`/`syscall`/`sys_*` references.
